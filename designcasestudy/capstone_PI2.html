Problem Statement

Problem: Polaris Intelligence’s current application already has many functionalities but it lacks a simple and straightforward way for companies to narrow down the scope of their analytics by selecting multiple variables.

Our Solution: Our module will allow Polaris’ customers to select and view data for two variables in a cross table for an aggregated selection of postal codes. This module will efficiently query and display searches in a few seconds while also storing previous queries for easy retrieval and recurrent review. Our scalable solution does not include changing Polaris' existing services, softwares, and data.

Stakeholders

Stakeholders in Polaris include Consumer Packaged Goods (CPG) companies, municipalities, educational institutions, and other small businesses.

Frontend Requirements:

  1. Drag and Drop functionality.
  2. Selection of two variables.
  3. Upload data in CSV form.
  4. View resulting crosstab.
  5. View previous crosstabs.
  6. Integratable into Polaris’ existing tech stacks.

Backend Requirements

  1. Efficiently query based on user input.
  2. Query for aggregated set of postal codes.
  3. Retrieve correct data from PostgreSQL.
  4. Calculate when necessary, e.g. convert %→count.
  5. Construct correct crosstab.
  6. Store past queries.

Full Process

The frontend uses HTML, Javascript, React, and CSS languages in php files. We used minimal HTML and CSS so that the Polaris team can easily integrate our frontend into the existing website. Javascript’s react-beautiful-dnd is used to create the drag and drop functionality for variable selection. POST is used to upload and store the selected user file. The final crosstab is visualized on the frontend using react-json-cable.
The backend solution uses PostgreSQL and PHP. After the user submits the selected variables, the variables are sent to the backend. First, the algorithm identifies the types of user selected variables and if the variable is a percentage, it calculates the variable count based on total count. Then, it creates a query statement corresponding to the variables and the types and queries the database. If the query is executed without errors and the resulting crosstab table is generated, the algorithm saves the result and stores the user selected variables to a table. After everything is generated and stored correctly, the generated crosstab table is sent to the frontend to show the user the desired crosstab table.

Performance

Using Apachebench, we tested the performance of our module with various amounts of concurrent requests to a maximum level of 40.

Based on the “Time per request” results, the worst case performance for each request from 40 concurrent users is handled in 840ms. The best performance is for one concurrent user request handled in 21ms. As such, our design is proven to meet the concurrent load of forty users and the performance requirements of less than two seconds.
